## 1. 도메인 정의 & 진입 모델

- **하단 네비**: `홈` / `피드` / `회원` — **모임 전용 탭은 없다.**
- **모임 도메인은 "탭"이 아니라 화면 묶음**이다. 진입은 항상 **홈이 넘겨준 모임**에서 시작한다.
  - 홈(메인) → 내 모임 선택 → **모임 홈(A3, 모임 도메인)**
  - 홈 검색결과/초대 링크/번개 공유 링크 → **모임 상세(비회원뷰, A1)** 또는 **번개 상세(A4)**
- **모임 도메인 범위**: 모임 상세·홈 · 정모/번개 · 정산 · 운영관리 · 알림/빈상태. **DM 제외.** 게시글·피드는 [피드 도메인 스펙](../피드/README.md)으로 분리(동일 담당).
- **제품 전제**: 모바일 앱 — **모든 사용자는 로그인 상태**(비로그인 플로우 없음). 딥링크 수신자가 앱 미설치면 스토어 유도.
- **행위자**: `부원(member)` / `모임장(owner — 슈퍼유저: 모임 생성·삭제 포함 모든 권한)` / `관리자(manager — 정모 주최·정산(총무 역할)·공지·가입 승인·강퇴 등 운영 실무)` / `번개 주최자(host)` / `비부원 참여자(공개 번개)`.

## 2. 핵심 설계 결정 — 기존 AI 와이어플로우(S1~S8 / W-01~47)와 무엇이 다른가

| 축 | 기존 AI판 | **본 설계(v6)** |
|---|---|---|
| 정보구조(IA) 축 | **돈 6단계** 중심 | **모임(커뮤니티) 실체** 중심 · 돈은 그 위에 얹음 |
| 정산 방식 | 앱 포인트 **에스크로 + 원장** | **카카오페이 송금요청**(외부) — KYC 부담으로 포인트 제거 |
| 정산 확인 | 원장 자동 대사 | 콜백 없음 → **참여자 셀프 신고 + 주최자 수동 확인**(2단계 기록) |
| 돈 모델 | 포인트 지갑 잔액 · 풀링 | **개설 시 정산 방식 택1(#10-1)** — 사후 정산(실지출 후 걷기) / 사전 걷기(예상 선납→차액 환급·추가정산). 둘 다 실지출 1/N(개인별 수정) · 풀링 없음(카카오페이 직접) |
| 가입폼 | 고정 가입폼 | **개설 시 선택: 즉시 가입 / 승인제(폼빌더)** |
| 정산 단위 | 차수(round) | **정모/번개(event)** · 정산 주체 = **이벤트 주최자** |
| 이벤트 개설 | 모임장만 | **정모=모임장·관리자** · **번개=부원 누구나** |
| 진입 | 모임 탭 | **홈에서 진입**(모임 전용 탭 없음) |

> **폐기 어휘**: `차수` · `에스크로` · `원장` · `포인트` · `KYC/지갑` · `공동회비풀`. ※ `사전 걷기/잉여 환급`은 **정산 방식 주최자 선택(#10-1)**으로 부활 — 사전 걷기 이벤트에서 유효. 정산은 **코디네이션(지출하고 · 걷고 · 확인하기)** 흐름이다.

```
모임(club, 지속) ⊃ 정모·번개(event, 1회성) ⊃ 정산(settlement, 이벤트 단위 사후 정산)
```

| 구분 | 정모 | 번개 |
|---|---|---|
| 개설 권한 | **모임장·관리자** | **부원 누구나** (모임 소속·주최 내역 기록) |
| 타입 | 단일 | **내부 번개**(부원만) / **공개 번개**(비부원 참여 = 가입 유입 경로) |
| 정산 주체 | **정모 주최자(모임장·관리자)** | **번개 주최자** |
| 모집 글 | **개설 시 자동 생성** `[크리틱4]` | **개설 시 자동 생성** — 공통 카드(제목·일시·장소·정원·참여 버튼) → 피드·게시판 노출 |

## 4. 도메인 경계 (다른 팀원과의 이음새)

- **← 회원 도메인**: 로그인 전제 · **팔로우 그래프·버튼·목록·프로필 소유**(나는 피드 노출만 소비) · 사용자 차단.
- **← 홈 도메인**: **내 모임 목록·진입 + 모임 탐색·검색 + 내 일정**(내가 참여 예정인 정모/번개 모아보기) 소유. 홈이 A1/A3/이벤트 상세로 넘겨준다.
- **⚠ 경계 확인 필요**: 게시글 신고/모더레이션(B3) ↔ 운영자 · 알림 인프라(X1) ↔ 공용/회원 · 팔로우 API ↔ 회원 .
- **⚠ W-05(홈, kyung 브랜치) 점검 결과(2026-07-09)**: 진입 구조(내 모임 카드→모임 진입 · 찾기/만들기 · 내 일정 캘린더 · "번개" 배지)는 본 스펙과 **일치** ✅. 단 **구 결제모델 잔재**가 남음 — 포인트 잔액 바 · "예치" 카드 · "출금 동의" 요청 · "동의 대기" 배지(→ 피벗 공유 필요), "정기모임" 표기(→ "정모"), 탭바의 DM 탭·중앙 FAB(3도메인 구도 밖, 규정 필요).
- **↔ 피드 도메인(자매 스펙, 동일 담당)**: 게시글·좋아요·댓글·신고·피드 노출은 [피드 스펙](../피드/README.md) 소유. 모임 홈의 게시판 탭 표면은 모임(A3), 게시글 시스템은 피드. 정모/번개 **모집 카드 생성은 모임(A4), 노출은 피드**.
- **소유 엔티티(모임)**: `모임(상세·홈·운영)` · `정모/번개(+모집 카드 생성)` · `멤버십` · `정산`.

## 5. 시나리오 (상세)

### 하위 도메인: 모임

#### 모임_둘러보기 — 비회원 진입 인덱스

**개요**
비부원이 모임을 발견해 미리보기를 열람하고 가입을 결정하기까지. 정보 제한 노출(잠금)로 **가입 전환**을 만드는 유입 경로의 입구. 유입원별로 진입 조건이 달라 하위 행위로 나눈다.

##### 모임_정보_열람 — 비회원 미리보기

**행위자**
비부원

**트리거 / 진입**
홈 탐색·검색 결과(홈 도메인) · 초대_링크_진입 · 공개 번개에서 "모임 보기" · 피드 카드의 모임명

**User Flow**
1. 모임 상세(비회원 뷰) 진입 — 커버·이름·카테고리·멤버 수·소개 전문
2. 활동 미리보기(최근 정모/번개 2~3건) 열람
3. 게시글·정산 기록은 🔒 잠금(부원 전용 안내)
4. 가입 방식 배지 확인(`즉시 가입`/`승인제`) → `[가입하기]` → 모임_가입_신청

**분기 / 엣지**
- 이미 부원 → 모임 홈으로 리다이렉트
- 승인 대기 중 → CTA `승인 대기` 비활성(철회는 모임_가입_취소)
- 해산된 모임 → "해산된 모임" 안내 + 탐색 유도
- **비부원 노출 범위(확정)** — 멤버는 인원수·모임장/관리자 역할 표시까지, 정산 아카이브는 🔒 잠금(가입 후 공개)

##### 초대_링크_진입 — 초대 딥링크로 유입

**행위자**
초대 링크 수신자(카톡 등 외부 채널)

**트리거 / 진입**
모임장·관리자가 생성·공유한 초대 링크 탭(초대_링크_관리에서 생성)

**User Flow**
1. 링크 탭 → 앱 열림(미설치 시 스토어 유도 — 정책 논의)
2. 모임 상세(비회원 뷰) 바로 진입 — 앱 사용자는 전원 로그인 상태(제품 전제)
3. 이후 모임_정보_열람과 동일

**분기 / 엣지**
- **회수된 링크** → 무효 안내 + 탐색 유도(만료 정책 확정: 무기한+수동 회수, 초대_링크_관리 참조)
- 이미 부원 → 모임 홈으로
- **초대자 표시(확정)** — 수신자 화면에 "OO님의 초대" 표시(신뢰 신호)

#### 모임_가입 — 가입 흐름 인덱스

**개요**
비부원이 부원이 되기까지의 전 과정. 모임 개설 시 선택된 **가입 방식(즉시 가입 / 승인제)**에 따라 흐름이 갈리고, 승인제는 관리자가 만든 가입폼을 중심으로 신청 → 심사(승인/거절) → (철회/재신청) 생애주기를 가진다.

**상태 전이**

```
미신청 ─[가입하기]─▶ (즉시 가입) ──────────────▶ 부원
   │
   └─(승인제)▶ 작성 중 ─제출▶ 승인 대기 ─┬─ 승인 ▶ 부원
                              │        ├─ 거절 ▶ 재신청 가능(정책)
                              │        └─ 철회(취소) ▶ 미신청
```

##### 모임_가입_폼_생성 — 가입폼 만들기·수정 (관리자 측)

**개요**
승인제 모임의 가입 질문 폼을 모임장·관리자가 직접 구성한다. 모임 개설 중 최초 생성되고, 운영 중에도 수정할 수 있다. 즉시 가입 모임에는 폼이 없다.

**행위자**
모임장 · 관리자

**트리거 / 진입**
- 모임 개설 중 가입 방식에서 **승인제 선택 직후**(모임_개설)
- 운영 중: 관리 메뉴에서 가입폼 수정 · 가입 방식을 즉시→승인제로 전환할 때(운영_관리)

**User Flow**
1. 가입폼 빌더 진입
2. 질문 추가 — 타입: 단답 / 장문 / 객관식(단일·복수), 최소셋 · **시스템 기본 필수 항목: 모임별 닉네임**(#9 — 모든 가입폼에 내장, 초대 경유자도 이것만은 작성)
3. 질문별 필수/선택 지정 · 드래그로 순서 변경 · 삭제
4. 신청자 화면 미리보기
5. 저장 → (개설 중이면) 개설 플로우 계속 / (운영 중이면) 즉시 반영

**분기 / 엣지**
- **폼 수정 시 대기 중 신청(확정)**: 제출 시점의 폼·응답 그대로 심사(구버전 유지) — 새 폼은 이후 신청부터 적용
- **최소 1질문 강제(확정)** — 승인제는 질문 1개 이상 필수(질문 0개 저장 불가)
- 질문 타입 확장 — MVP 제외(확정), 단답·장문·객관식 3타입 유지

##### 모임_가입_신청 — 가입하기 (신청자 측)

**개요**
비부원이 모임 상세(비회원 뷰)에서 가입을 시작하는 흐름. 즉시 가입이면 한 번의 탭으로 부원이 되고, 승인제면 폼을 작성·제출해 승인 대기 상태가 된다.

**행위자**
비부원

**트리거 / 진입**
모임 상세(비회원 뷰)의 `[가입하기]` — 유입: 홈 탐색·초대 링크·공개 번개(가입 유입 경로)

**User Flow**
**즉시 가입**
1. `[가입하기]` 탭 → **모임별 닉네임 1회 입력**(전역 닉네임 프리필 — 크리틱5 ②) → 즉시 부원 편입 → 모임 홈 랜딩(환영 상태)

**승인제**
1. `[가입하기]` → 가입폼 렌더링(관리자가 구성한 질문)
2. 응답 작성(필수 항목 검증) → 제출
3. `신청 완료 / 승인 대기` 상태 표시 → 결과는 알림으로 수신

**분기 / 엣지**
- **중복 신청 차단**: 이미 승인 대기 중이면 CTA `승인 대기` 비활성
- **제출 후 응답 수정 불가(확정)** — 고치려면 철회(모임_가입_취소) 후 재신청으로 갈음
- **강퇴 이력자(확정)**: 재가입 시 승인제 경로 강제(멤버_강퇴 참조)
- **블랙리스트 등록자(신설)**: 가입 신청 자체 차단 — 버튼 비활성 + `이 모임에 가입할 수 없어요` 중립 문구(사유 미표기)

##### 모임_가입_승인 — 심사·승인 (관리자 측)

**개요**
모임장·관리자가 대기 중 신청의 폼 응답을 열람하고 승인해 부원으로 편입시키는 흐름.

**행위자**
모임장 · 관리자

**트리거 / 진입**
- 가입 신청 발생 알림
- 관리 메뉴의 가입 신청 목록(대기 N건)

**User Flow**
1. 가입 신청 목록(신청일 순) 진입
2. 신청자 선택 → 폼 응답 전체 열람 `[크리틱2]`
3. `[승인]` → 신청자에게 승인 알림 → 부원 편입 완료
4. 신청자는 알림에서 모임 홈으로 진입

**분기 / 엣지**
- **승인 vs 철회 경쟁 상태**: 승인 처리 순간 신청자가 철회하면 → 철회 우선(부원 미편입), 관리자에게 안내(모임_가입_취소와 연동)
- **일괄 승인 지원(확정)** — 체크박스 다중 선택 → 일괄 승인(개별 응답 열람은 여전히 가능)
- **처리 이력 보관(확정)** — 누가 언제 승인/거절했는지 기록, 모임장·관리자만 열람

##### 모임_가입_거절 — 거절·재신청 (관리자 → 신청자)

**개요**
심사에서 거절하고 사유를 전달하는 흐름과, 거절된 신청자가 재신청하는 생애주기.

**행위자**
모임장·관리자(거절) · 신청자(재신청)

**트리거 / 진입**
가입 신청 목록의 응답 열람 → `[거절]`

**User Flow**
**거절(관리자)**
1. 응답 열람 → `[거절]` → 사유 입력(선택)
2. 신청자에게 거절 알림(사유 포함 여부는 입력 시)

**재신청(신청자)**
1. 거절 안내 확인 → 재신청 가능 시점 안내
2. 거절 7일 후 재신청 가능 — 모임_가입_신청 흐름 재진입(폼 재작성)

**분기 / 엣지**
- **재신청 정책(확정)**: 거절 후 **쿨다운 7일**, 횟수 제한 없음 — 반복 신청 피로감 방지
- **재신청 프리필(확정)** — 이전 응답 미리 채움(폼이 바뀌었으면 새 질문만 빈칸)
- 사유 미입력 거절의 표시 문구 — 📝카피(화면 구현 단계에서 확정)

##### 모임_가입_취소 — 신청 철회 (신청자 측)

**개요**
승인 대기 중인 신청자가 스스로 신청을 거두는 흐름. 기존 "논의" 항목에서 하위 행위로 승격 — 철회를 허용한다.

**행위자**
신청자(승인 대기 상태)

**트리거 / 진입**
- 모임 상세(비회원 뷰)의 `승인 대기` 상태 표시 → 신청 상태 확인
- 알림/내 활동에서 대기 중 신청 확인(회원 도메인 경계 확인 필요)

**User Flow**
1. 대기 중 신청 확인 → `[신청 철회]`
2. 확인 다이얼로그(작성한 응답이 삭제됨 고지) → 철회 확정
3. 상태 `미신청` 복귀 — 관리자 신청 목록에서 제거
4. `[가입하기]` CTA 재활성

**분기 / 엣지**
- **철회 vs 승인 경쟁 상태**: 관리자가 승인하는 순간 철회 발생 → 철회 우선, 관리자에게 "철회됨" 안내(모임_가입_승인과 연동)
- **철회 후 즉시 재신청 허용(확정)** — 본인 의사 철회라 쿨다운 없음(거절 쿨다운 7일과 구분)
- 철회 이력도 처리 이력에 포함(확정) — 관리자만 열람(모임_가입_승인 참조)

#### 모임_홈 — 회원 허브 인덱스

**개요**
부원의 활동 허브. 요약 열람과 과거 기록(아카이브) 열람으로 나뉜다. 지난 이벤트+정산 아카이브 진입이 투명성의 핵심.

##### 모임_홈_열람 — 허브 요약·분기

**행위자**
부원 · 모임장·관리자(관리 진입 추가 노출)

**트리거 / 진입**
홈의 내 모임 카드(홈 도메인 핸드오프) · 피드 카드 · 알림 딥링크 · 가입 완료 직후 랜딩

**User Flow**
1. 모임 홈 진입 — 커버·역할 칩
2. 공지 카드(NEW) → 게시글 상세(피드 도메인)
3. 다가오는 정모/번개 → 이벤트 상세(정모_번개_참여)
4. 멤버 요약 · 최근 게시글 → 게시판(피드 도메인)
5. (모임장·관리자) `관리` 진입(운영_관리) — RBAC 분기

**분기 / 엣지**
- 빈 상태: 신규 모임(공지·이벤트·게시글 0) → 역할별 첫 액션 유도(공통/빈_상태)
- 내 모임 프로필: **모임별 닉네임 수정**은 모임 내 설정에서(크리틱5 ④) — 정산·명단·채팅은 항상 현재 닉네임으로 통일 표시
- 진행 중 정산 배너 — 📝UI 회의로 이연(#7 결정 5)

##### 지난_이벤트_열람 — 과거 기록·투명성

**행위자**
부원 전체(비부원 불가 — 미리보기에서 잠금)

**트리거 / 진입**
모임 홈 "지난 이벤트"

**User Flow**
1. 종료 정모/번개 목록(일시·장소·참여 수·정산 여부 배지)
2. 이벤트 선택 → 종료 상태 상세
3. 정산 완료 이벤트 → **정산 기록 열람**(정산_마감/아카이브_열람로 연결)

**분기 / 엣지**
- **`정산 없음` 명시(확정)** — 기록 부재와 무정산을 구분(투명성 일관)
- **무한 스크롤 + 주최자 필터(#9 확정)** — 보관은 영구(투명성). 주최자 기반 필터로 "누가 연 이벤트"를 모아볼 수 있음(주최 이력 신호와 연동)
- 취소된 이벤트 포함(확정) — `취소됨` 배지(이벤트_취소 참조)

#### 모임_개설 — 모임 생애주기 인덱스 (생성~해산)

**개요**
모임의 생성과 소멸. 생성 시 가입 방식(즉시/승인제)을 선택하며, 개설자가 **모임장**이 된다. 생성·삭제(해산)는 모임장 전용 권한.

##### 모임_생성 — 새 모임 만들기

**행위자**
사용자 누구나 — 완료 시 **모임장**이 된다

**트리거 / 진입**
홈 "모임 만들기"(홈 도메인 핸드오프 — FAB 액션시트 여부는 경계 확인)

**User Flow**
1. 이름·소개·카테고리·커버 입력
2. **가입 방식 선택**: 즉시 가입 / 승인제 `[크리틱3]`
3. (승인제) 가입폼 구성 — 모임_가입_폼_생성으로 연결
4. 개설 완료 → 초대 링크 즉시 공유 제안 → 모임 홈 랜딩

**분기 / 엣지**
- **첫 액션 = 부원 초대 우선(확정)** — 완료 화면에서 초대 링크 공유 첫 제안, 정모 만들기는 두 번째
- **커버 기본 이미지 세트 제공(확정)** — 카테고리별 기본 커버(디자인 시스템 톤)
- **동일 이름 불가(#9 확정)** — 모임 이름 유일 강제, 개설·수정 시 중복 검사

##### 모임_정보_수정 — 프로필·가입 방식 변경

**행위자**
모임장 (v10: 정보 수정은 모임장 전용)

**트리거 / 진입**
운영_관리 → 모임 설정

**User Flow**
1. 프로필(이름·소개·커버·카테고리) 수정 → 저장
2. 가입 방식 전환(즉시↔승인제) — 승인제 전환 시 모임_가입_폼_생성 연결

**분기 / 엣지**
- **승인제→즉시 전환 시(확정)**: 대기 중 신청 전원 **자동 승인** + 알림(문턱을 없앴다는 뜻이므로 일관 처리)
- 이름 변경 시 초대 링크·모집 카드의 표기 갱신
- 수정 이력 보관 — MVP 제외(확정), 정산 기록만 append 원칙 적용

##### 모임_해산 — 모임 삭제

**행위자**
모임장만 (v10: 슈퍼유저 전용)

**트리거 / 진입**
운영_관리 → 모임 설정 → 위험 구역 "모임 해산"

**User Flow**
1. 해산 선택 → 영향 고지(부원 N명·게시글·아카이브)
2. **미완 정산 존재 시 차단** — 정산 마감 후 재시도 안내
3. 2단계 확인(모임 이름 입력 등) → 해산 확정
4. 전 부원 알림 → 홈으로

**분기 / 엣지**
- **해산 후 읽기 전용 보존(확정)** — 전 부원이 게시글·정산 아카이브 열람 가능("해산으로 증거 인멸" 방지)
- 해산된 모임 링크 접근 → "해산된 모임" 안내(모임_정보_열람 연동)
- **1인 모임 해산 간소화(확정)** — 영향자 없으므로 이름 입력 생략, 단순 확인만

#### 모임_채팅 — 상시 채팅방 (leaf)

**개요**
모임당 1개의 상시 채팅방 — 이벤트와 무관한 평시 소통. "단톡방 대체" 스토리의 완성. 채팅은 2층 구조: **상시방(모임당 1) + 이벤트방(정모/번개당 1, 정모_번개_채팅)**.

**행위자**
부원 전원(가입 시 자동 입장, 탈퇴·강퇴 시 퇴장) · 공지 고정은 모임장·관리자

**User Flow**
1. 모임 생성 시 자동 생성 → 부원 자동 입장
2. 대화(텍스트·사진) · 표시명 = 모임별 닉네임
3. 모임 해산 → 읽기 전용 아카이브(해산 보존 원칙과 일치)

**분기 / 엣지**
- 탈퇴·강퇴자 메시지는 실명+`나간 멤버` 표시 유지(크리틱 ⑧)
- 채팅 알림 카테고리(알림_설정 연동, 논의) · 채팅 내 신고(⚠팀회의 차단·신고와 연동)

#### 운영_관리 — 관리자 실무 인덱스

**개요**
모임장·관리자의 일상 운영. 가입 심사는 모임_가입(승인·거절), 정모 개설은 정모_번개로 분리 — 여기는 공지·초대·역할.

**역할 권한 (v10 확정)**
모임장 = 슈퍼유저(전권) / 관리자 = 정모 주최·정산·공지·가입 승인·강퇴 등 운영 실무. 해산·승계·정보 수정은 모임장 전용 — **강퇴는 관리자도 가능(2차 검증 확정)**.

##### 공지_관리 — 공지 작성·수정·삭제

**행위자**
모임장 · 관리자

**트리거 / 진입**
관리 메뉴 "공지" · 게시판에서 공지 모드 작성

**User Flow**
1. 공지 작성(텍스트·사진) — 글 시스템은 피드 도메인, **공지 권한만 모임 소유**
2. 게시 → 게시판 상단 고정 + 전 부원 알림
3. 수정·삭제 · 고정 해제(오래된 공지)

**분기 / 엣지**
- **공지 무제한 · 최신순(#9 확정)** — 개수 제한 없음, 고정 정렬은 최신순
- 공지에 댓글 허용 여부(피드 도메인과 협의)
- **공지=자유 글(확정)** — 시스템 알림과 별개, 중복 허용(과도한 규칙화 배제)

##### 초대_링크_관리 — 생성·공유·회수

**행위자**
모임장 · 관리자

**트리거 / 진입**
관리 메뉴 "초대 링크" · 모임_생성 완료 직후 공유 제안

**User Flow**
1. 링크 생성 → 복사·카톡 공유
2. 수신자: 초대_링크_진입(모임_둘러보기) → 가입 흐름
3. 링크 회수(무효화)·재발급

**분기 / 엣지**
- **만료 정책(확정)**: 기본 무기한 + 관리자 수동 회수·재발급 — 카톡 링크 공유 관행과 일치
- **초대 경유 배지(확정)** — 신청 목록에 `초대 경유` 표시(심사 참고)
- **초대 경유자 처리(#9 확정)** — 폼 생략 불가하되 **필수 항목(모임별 닉네임 등)만** 작성(부가 항목은 원래 optional이라 생략 가능), **승인 과정 생략 → 제출 즉시 부원**(초대 = 이미 검증됨)

##### 역할_관리 — 관리자 지정·해제

**행위자**
모임장만 (역할 변경 권한)

**트리거 / 진입**
관리 메뉴 → 멤버 목록 → 대상 선택

**User Flow**
1. 멤버 목록(역할 칩: 모임장/관리자/부원)
2. 부원 선택 → `관리자로 지정` → 확인 → 대상자 알림
3. 관리자 선택 → `관리자 해제` → 부원으로

**분기 / 엣지**
- **관리자 수 제한 없음(확정)** — 모임장 재량
- **해제돼도 주최자 자격 유지(확정)** — 정산 권한은 역할이 아닌 "주최 행위"에 귀속, 마감까지 본인 책임(번개 주최 부원과 동일 원칙)
- 모임장 이전은 별도 흐름(탈퇴_강퇴_승계/모임장_승계)

##### 블랙리스트_관리 — 재가입 차단 목록

**개요**
특정 사용자의 재가입을 원천 차단하는 목록. 강퇴(승인제 강제)만으로는 "실수로 다시 승인"할 수 있어, 그 실수를 구조적으로 막는 상위 단계. 강퇴 순간뿐 아니라 **사후 등록**도 가능(B안) — 먼저 탈퇴해버린 문제 멤버도 커버.

**행위자**
등록: 모임장·관리자 · 해제: 모임장·관리자 (공동 결정: 둘 다 관리자 가능)

**트리거 / 진입**
- 강퇴 확인 다이얼로그의 `블랙리스트에도 등록` 체크(멤버_강퇴)
- 관리 메뉴 → 블랙리스트 → `나간 멤버 이력`(강퇴·탈퇴자, 처리 이력 데이터 재사용)에서 선택 등록

**User Flow**
1. 목록 열람 — 등록자·사유·등록일·등록 처리자
2. 등록: 강퇴 시 체크 또는 나간 멤버 이력에서 선택 → 사유 입력(선택) → 확정
3. 해제: 대상 선택 → 확인 → 즉시 가입 신청 가능 상태로 복귀

**효과 (재가입 규칙과의 관계)**
| 이력 | 재가입 |
|---|---|
| 일반 탈퇴 | 자유(즉시 가입 모임은 바로) |
| 강퇴(블랙 미등록) | 승인제 경로 강제(기존 확정 유지) |
| **블랙리스트** | **가입 신청 자체 차단** — 초대 링크·공개 번개 경유 포함 |

**분기 / 엣지**
- 차단 표시: 대상자에게 `이 모임에 가입할 수 없어요` 중립 문구 + 버튼 비활성(사유 미표기 — 공동 결정)
- 블랙리스트 등록자의 공개 번개 참여도 차단(가입 유입 경로 봉쇄 일관성 — 제안, 논의)
- 다른 모임과 비공유(강퇴 이력 비노출 확정과 동일 원칙)
- 대상자 알림 여부(비알림 권장 — 마찰 최소화, 논의)

#### 탈퇴_강퇴_승계 — 멤버십 종료·권한 이전 인덱스

**개요**
멤버십의 종료(자발 탈퇴/강퇴)와 모임장 권한 이전. 미완 정산이 걸린 채 사람이 빠지는 상황(orphan)을 구조적으로 방지한다.

##### 모임_탈퇴 — 스스로 나가기

**행위자**
부원 · 관리자 (모임장은 승계 없이는 탈퇴 불가 → 모임장_승계)

**트리거 / 진입**
모임 홈 ⋮ 메뉴 → "모임 나가기"

**User Flow**
1. 나가기 선택 → 확인 다이얼로그
2. **미납 정산 있으면 고지** `[크리틱4]` — 탈퇴해도 정산 명단에는 유지(주최자가 조정)
3. **내가 주최한 진행 중 정산 있으면** → 모임장에게 이관 안내 후 진행 `[크리틱4]`
4. 확정 → 멤버십 종료 → 홈으로

**분기 / 엣지**
- **탈퇴자 기록 실명 유지(확정)** — 게시글·정산 기록 그대로, `탈퇴한 멤버` 표시만 추가(아카이브 무결성)
- 탈퇴 후 재가입: 즉시 가입 모임은 바로, 승인제는 새 신청(강퇴와 달리 제한 없음)
- 참여 예정 이벤트가 있으면 → 자동 참여 취소 + 주최자 알림

##### 멤버_강퇴 — 제명

**행위자**
모임장 · **관리자**(2차 검증 확정 — 강퇴 위임)

**트리거 / 진입**
운영_관리 → 멤버 목록 → 대상 선택 → "강퇴"

**User Flow**
1. 대상 선택 → 사유 입력 → 확인 다이얼로그(+`블랙리스트에도 등록` 체크 옵션 — 블랙리스트_관리 참조)
2. 미납 정산 있으면 고지(명단 유지) `[크리틱4]`
3. 확정 → 제명 → 대상자에게 알림(사유 포함)

**분기 / 엣지**
- **강퇴 이력자 재가입(확정)**: 허용하되 **승인제 경로 강제**. 단 **블랙리스트 등록자는 가입 신청 자체 차단**(블랙리스트_관리 — 강퇴 시 체크 또는 사후 등록)
- 강퇴 대상이 관리자면 → 역할 해제 후 강퇴(자동 처리)
- 강퇴자의 게시글·참여 기록 처리(탈퇴와 동일 정책)
- **강퇴 이력 타 모임 비노출(확정)** — 모임 간 이력 공유 없음(낙인 방지)

##### 모임장_승계 — 소유권 이전

**행위자**
모임장(위임) · 지정 대상(수락 여부 논의)

**트리거 / 진입**
- 자발 위임: 운영_관리 → 멤버 → "모임장 위임"
- **강제**: 유일 모임장이 탈퇴 시도 → 승계 지정 없이는 탈퇴 불가

**User Flow**
1. 대상 선택(부원·관리자 중) → 확인(권한 전부 이전 고지)
2. 확정 → 역할 전환(기존 모임장은 부원 또는 관리자로 — 선택)
3. 대상자·전 부원 알림

**분기 / 엣지**
- **대상자 수락 절차** 필요 여부(일방 지정 vs 수락 후 확정, 논의)
- 승계와 동시에 진행 중이던 모임장 주최 정산 → 신 모임장에게 이관
- **잠수 모임장 승계(#9 확정)** — 관리자가 승계 요청 → 모임장에게 **1주 간격으로 3회 알림** → 3회 모두 무응답 시 요청한 관리자에게 **자동 승계**(운영자 개입 불필요, 총 3주 프로세스)

### 하위 도메인: 정모_번개

#### 정모_번개_개설 — 이벤트 만들기·모집 인덱스

**개요**
이벤트(정모/번개) 생성과 모집 확산. 완료 시 **모집 카드가 자동 생성**되어 게시판·피드에 노출된다(생성=모임, 노출=피드).

##### 정모_개설 — 정기 이벤트 만들기

**행위자**
모임장 · 관리자 (v10 확정)

**트리거 / 진입**
운영 관리 메뉴 "정모 개설" · 모임 홈 이벤트 섹션(권한자에게 노출)

**User Flow**
1. 제목·일시·장소·정원(선택)·설명 입력 + **신청 마감 기한**(이후 참여·취소 불가) + **정산 방식 선택**(사후 정산 / 사전 걷기 — 사전이면 예상 회비 입력, #10-1) + **노쇼 페널티 비용**(선택, #8 확정)
2. 완료 → **모집 카드 자동 생성** → 게시판·피드 노출 + 부원 알림
3. 완료 화면에서 카드 미리보기 + 공유 제안(이벤트_공유)

**분기 / 엣지**
- **반복 정모 MVP 제외(확정)** — 단건 생성만, 반복은 애자일 백로그
- **예상 비용 선택 필드(확정)** — 참고 정보로 제공, 정산과 무관함을 명시
- 정모는 부원만 참여(공개 타입 없음 — 공개 모집은 번개 소관)

##### 번개_개설 — 즉흥 이벤트 만들기

**행위자**
부원 누구나 (모임 소속 · 주최 내역 기록)

**트리거 / 진입**
모임 홈/게시판 "번개 만들기"

**User Flow**
1. 제목·일시·장소·정원(선택) 입력 + **신청 마감 기한** + **정산 방식 선택**(사후 정산 / 사전 걷기 — 사전이면 예상 회비 입력, #10-1) + **노쇼 페널티 비용**(선택, #8 확정)
2. **타입 선택**: 내부 번개(부원만) / 공개 번개(비부원 참여 = 가입 유입 경로)
3. 완료 → **모집 카드 자동 생성** `[크리틱3]` → 게시판·피드(+팔로워 피드) 노출
4. 주최 내역 기록 — 주최자가 이후 정산 책임(정산_만들기)

**분기 / 엣지**
- 공개 번개의 노출 범위(팔로워 피드 + 공유 링크 — 홈 탐색 노출은 홈 도메인 협의)
- **개설 빈도 제한 없음(확정)** — 문제 발생 시 강퇴·신고로 대응, 제한은 필요해질 때
- **주최 이력 수 표시(확정)** — 이벤트 상세에 "주최 N회"(긍정 신호만, 부정 신호는 취소 이벤트의 자연 노출로 충분)

##### 이벤트_공유 — 딥링크 확산

**행위자**
주최자(주로) · 참여 부원(공유 허용 여부 논의)

**트리거 / 진입**
이벤트 상세·개설 완료 화면의 공유 버튼

**User Flow**
1. 공유 → 딥링크 생성 → 카톡 등 외부 채널로 전달
2. 수신자: 링크 탭 → 이벤트 상세 진입(앱 사용자는 전원 로그인 상태 — 미설치자는 스토어 유도)
3. 내부 번개/정모 링크를 비부원이 열면 → 모임 상세(비회원 뷰)로 우회 + 안내

**분기 / 엣지**
- **내부 링크 비부원 = 모임 소개로 우회(#8 확정)** — 이벤트 내용은 숨기고 "부원 전용" 안내 + 가입 유입 경로로 전환
- 링크 만료(이벤트 종료 후) → 종료 안내
- 공유 링크 유입 추적 — MVP 제외(확정), 성장 단계 백로그

#### 정모_번개_참여 — 참여 흐름 인덱스

**개요**
참여자 관점의 이벤트 흐름: 참여 → (취소) → 종료 후 정산 참여(회비_납부). 공개 번개는 비부원 참여가 가능해 별도 행위로 둔다.

**상태 전이 (참여자 관점)**
```
미참여 ─[참여]─▶ 참여 확정 ─┬─ [취소] ▶ 미참여
                          ├─ 이벤트 취소됨 ▶ 해제(알림)
                          └─ 이벤트 종료 ▶ 정산 참여자(회비_납부) · 노쇼면 명단 조정
```

##### 이벤트_참여 — 정모/내부 번개 참여 (부원)

**행위자**
부원

**트리거 / 진입**
모임 홈 이벤트 리스트 · 피드 모집 카드 · 홈 일정(홈 도메인) · 알림 · 공유 링크

**User Flow**
1. 이벤트 상세 — 일시·장소·주최자·참여자/정원 게이지·타입 배지
2. `[참여]` → 명단 반영 → 참여 확정 표시
3. (종료 후 정산 생성 시) 정산 참여자로 자동 포함 → 회비_납부

**분기 / 엣지**
- **정원 마감** → 참여 비활성 + `마감` 배지(대기열 없음 — 단순 유지)
- **지각 합류 허용(확정)** — 종료 전까지 참여 버튼 유지, 정산 명단에도 포함(번개 문화상 중도 합류 자연스러움)
- **참여자 명단 부원 간 공개(확정)** — 누가 오는지 보고 참여 결정하는 문화 반영

##### 공개_번개_참여 — 비부원 참여·가입 유입 경로

**행위자**
비부원(멤버십 불필요 — 앱 사용자면 누구나)

**트리거 / 진입**
팔로우 피드의 번개모집 카드 · 공유 링크

**User Flow**
1. 공개 번개 상세(제한 뷰) — 일시·장소·정원·주최자
2. `[참여]` — 멤버십 없이 즉시 가능
3. 참여 후 **모임 상세(비회원 뷰) 노출 = 가입 유입 경로**
4. 종료 후 정산 참여자로 포함 — 비부원도 동일 납부 흐름(회비_납부) `[크리틱3]`

**분기 / 엣지**
- **비부원 노출 범위(확정)** — 인원수·주최자만, 부원 명단은 가입 후 공개
- 참여한 비부원의 모임 게시판·좋아요 권한(피드 도메인 협의)
- **노쇼 비부원 재참여(확정)** — 시스템 제한 없음, 노쇼 기록은 주최자에게만 보임

##### 참여_취소 — 참여 철회

**행위자**
참여 확정자(부원·공개 번개 비부원)

**트리거 / 진입**
이벤트 상세의 `참여 취소`

**User Flow**
1. 참여 취소 → 확인 → 명단 제외
2. 정원 마감 상태였다면 → 마감 해제(빈자리 발생)
3. 주최자에게 취소 알림(선택 — 소음 조절 논의)

**분기 / 엣지**
- **신청 마감 기한제(#8 확정)** — 주최자가 개설 시 설정한 **신청 마감 기한**까지 자유 취소. 기한 후 취소·불참은 노쇼로 간주 → **노쇼 페널티 비용**(개설 시 별도 설정) 적용(노쇼_처리 참조)
- 마감 후 취소 = 노쇼와의 경계(주최자가 노쇼_처리에서 판단)
- 사후 정산 모델이라 취소 시 돈 이동 없음(정산은 종료 후 실참여 기준)

#### 정모_번개_관리 — 주최자 관리 인덱스

**개요**
주최자 관점의 이벤트 운영: 정보 수정, 취소, 종료 후 노쇼 처리. 처리 결과가 정산 명단(정산_만들기)에 직결된다.

##### 이벤트_정보_수정 — 일시·장소 변경

**행위자**
주최자(정모=개설한 모임장·관리자, 번개=주최 부원)

**트리거 / 진입**
이벤트 상세(주최자 뷰) → 수정

**User Flow**
1. 제목·일시·장소·정원·설명 수정
2. 저장 → **전 참여자에게 변경 알림**(변경 항목 명시)
3. 모집 카드 내용 자동 갱신(게시판·피드)

**분기 / 엣지**
- **정원 축소 하한(#8 확정)** — 현재 참여자 수 미만으로는 축소 입력 자체가 불가(초과 상태가 생길 수 없음)
- **일시 변경 시 재확인 요구(확정)** — 날짜가 바뀌면 기존 참여 리셋 + 전원 재확인 알림(사실상 재모집)
- **번개 타입 변경 허용(#8 확정)** — 내부↔공개 토글 + 전 참여자 알림. **공개→내부 전환 시 기참여 비부원은 유지**(크리틱5 ⑦ — 정원 하한과 같은 기득권 보호, 신규만 부원 제한)

##### 이벤트_취소 — 주최자 취소

**행위자**
주최자

**트리거 / 진입**
이벤트 상세(주최자 뷰) → 취소

**User Flow**
1. 취소 선택 → 사유 입력(선택) → 확인
2. 전 참여자 알림 → 이벤트 상태 `취소됨`
3. 모집 카드 상태 갱신(참여 버튼 비활성)

**분기 / 엣지**
- **사후 정산 모델이라 돈 이동 없음**(정산 생성 전 취소가 일반) — 정산 생성 후라면 정산_취소(정산_만들기)와 연동
- **취소 이벤트 기록 포함(확정)** — 지난 이벤트 목록에 `취소됨` 배지로 표시(기록 은폐 없음, 반복 취소 패턴도 자연 노출)
- 반복 취소 신뢰 신호(확정) — 별도 표시 없음, 취소 이벤트가 목록에 남아 자연 노출(번개_개설의 주최 이력 참조)

##### 노쇼_처리 — 실참여 명단 확정

**행위자**
주최자

**트리거 / 진입**
이벤트 종료 후 상세(주최자 뷰) · 정산_생성의 참여자 확정 단계

**User Flow**
1. 참여 확정자 명단 → 불참자 노쇼 체크
2. 실참여 명단 확정 → **정산 참여자 기준으로 반영**(정산_생성)
3. (선택) 뒤늦은 정정 — 정산 생성 전까지 수정 가능

**분기 / 엣지**
- **노쇼 페널티(#8 · 옵션)** — 주최자가 개설 시 설정한 경우에만. 노쇼는 페널티 금액으로 정산 명단에 포함(전체 공개). **페널티 수입은 실참여자 분담금에서 차감**해 참여자 부담을 줄여줌(크리틱5 ① 검증식). **사후 정산**=노쇼가 페널티를 사후 납부 · **사전 걷기**=노쇼 선납분에서 **페널티만 차감**(나머지 환급) — 이미 받아둬 수금 확실(#10-1)
- 노쇼인데 일부 비용 분담해야 하는 경우(예약금 성격) → 개인별 금액 수정으로 처리(정산_생성)
- 마감 후 취소자와 노쇼의 구분 기준(참여_취소 연동)

#### 정모_번개_채팅 — 이벤트 채팅방 (leaf)

**개요**
정모/번개마다 붙는 채팅방. **개설 시 자동 생성, 참여 확정 시 자동 입장** — 명단과 채팅방 멤버가 항상 일치(공개 번개 비부원 포함). 인프라는 개인 채팅(회원 도메인)과 공유.

**행위자**
이벤트 참여자 전원(비부원 포함) · 주최자(방장 격 — 공지 고정 등, 논의)

**트리거 / 진입**
이벤트 상세의 채팅 탭/버튼 · 채팅 목록(회원 도메인 표면과 공유, 논의)

**User Flow**
1. 이벤트 개설 → 채팅방 자동 생성(주최자 입장)
2. 참여 확정 → 자동 입장 + 입장 안내 · 참여 취소 → 자동 퇴장
3. 대화(텍스트·사진 — 범위 논의) · 정산 생성/리마인드 등 시스템 메시지 삽입(논의)
4. 이벤트 종료 → **정산 마감까지 활성 유지**(정산 소통 채널) → 마감 후 **읽기 전용 아카이브**

**분기 / 엣지**
- 노쇼자의 채팅방 잔류 여부(정산 참여자이므로 마감까지 유지 권장, 논의)
- **일시 변경(참여 리셋) 시 채팅방 유지**(크리틱5 ⑧ — 대화 맥락 보존·재참여 유도 채널) · 탈퇴·강퇴자 메시지는 실명+`나간 멤버` 표시
- 이벤트 취소 시 → 즉시 읽기 전용(기록 보존 원칙)
- 채팅 내 신고는 사용자 신고 축(⚠팀회의 — 차단·신고 항목과 연동)

### 하위 도메인: 정산

#### 정산_만들기 — 정산 생성·변경 인덱스 (주최자)

**개요**
이벤트 정산의 생성·수정·취소. **정산 방식 2종(#10-1, 개설 시 택1)** — **사후 정산**: 이벤트 후 실지출 기준 걷기(환급·추가정산 없음). **사전 걷기**: 예상 회비 선납 → 이벤트 후 실지출로 정산 → 잉여 **환급** / 부족분 **추가정산**. 두 방식 다 정산 자체는 실지출 기준 1/N(개인 조정 가능), 이벤트당 정산 1건. 앱은 돈을 거치지 않고 기록·확인만(카카오페이 person-to-person).

##### 정산_생성 — 새 정산 요청

**행위자**
이벤트 주최자(정모=개설한 모임장·관리자, 번개=주최 부원)

**트리거 / 진입**
이벤트 종료 후 상세의 `정산하기` · (사전 걷기) 개설 직후 예상 회비 **선납 요청**도 발생

**User Flow (공통)**
1. **실지출 항목 입력**(항목명·금액) + **영수증 사진 선택 첨부** `[크리틱4]`
2. 참여자 확정 — 실참여 명단 자동 로드(노쇼_처리 반영) · **노쇼자는 페널티 비용으로 명단 포함**(#8 — 설정된 경우, 기록 전체 노출)
3. **1/N 자동 계산 → 개인별 금액 수정** `[크리틱2]` (예: 미음주자 감액)
4. **납부 기한 설정** `[크리틱3]`
5. 요약 미리보기 → 요청 생성 → 참여자 알림 + 링크 공유

**방식별 분기 (#10-1)**
- **사후 정산**: 5 이후 참여자가 실지출 몫을 납부(회비_납부)
- **사전 걷기**: 개설~참여 시 예상 회비를 **선납**(회비_납부 재사용) → 정산에서 앱이 **(선납−실지출) 차액** 산출 → 잉여=**환급**(환급_수령확인) / 부족=**추가정산**(추가정산_송금_진입) · 노쇼 페널티(설정 시)는 선납서 **차감해 실참여자 분담 경감**, 노쇼자 나머지는 환급

**분기 / 엣지**
- **합계 일치 강제(#7 + 크리틱5 ①)** — 검증식: **실참여자 분담 합계 = 실지출 − 노쇼 페널티 수입**. 사후=이 금액을 걷음 / 사전=선납 총액과 대조해 차액 환급·추가(걷는 총액은 항상 실지출과 일치)
- **영수증 없는 항목 표시(확정)** — `영수증 없음` 중립 표시로 검증 여부를 한눈에
- 비부원 참여자 포함 시 알림 도달 확인(앱 계정 전제)
- 참여자 0명(전원 노쇼) 엣지 → 정산 생성 불가 안내

##### 정산_수정 — 금액·명단 정정

**행위자**
주최자 (트리거는 자발 정정 또는 이의_처리의 인정)

**트리거 / 진입**
납부 현황판 → 수정 · 이의_처리에서 인정 시

**User Flow**
1. 실지출 항목·개인별 금액·기한·명단 수정
2. 저장 → **전 참여자에게 변경 알림**(변경 내용 명시)
3. **이미 납부한 참여자의 금액이 줄었다면** → 차액을 주최자가 카카오페이로 환불(앱은 금액·완료 체크만) `[크리틱4]`
4. 금액이 늘었다면 → 추가 납부 요청 상태로 전환

**분기 / 엣지**
- **수정 이력 기록(확정)** — 몇 차 정정인지 표시, 아카이브에 이력 노출(아카이브_열람 참조)
- 납부 `확인` 완료자의 금액 수정 → 해당자만 상태 재오픈
- 마감 후 수정 불가(확정) — 오류는 정정 기록 추가로(마감_확정 참조)

##### 정산_취소 — 정산 철회

**행위자**
주최자

**트리거 / 진입**
납부 현황판 → 정산 취소

**User Flow**
1. 취소 선택 → 영향 고지(이미 납부 N명)
2. 확인 → 정산 무효화 → 전 참여자 알림
3. **납부분 전액 환불** — 주최자가 카카오페이로 개별 송금, 앱은 환불 체크리스트만 `[크리틱4]`
4. 환불 완료 체크 후 종결

**분기 / 엣지**
- 취소 후 재생성 허용(같은 이벤트에 새 정산 — 이벤트당 1건 원칙과 정합: 취소본은 무효 처리)
- 환불 미완 상태로 방치 시 리마인드(참여자 측 이의 경로 유지)
- **취소 사유(확정)** — 입력 선택, 입력 시 참여자 알림·기록에 노출

#### 납부_추적 — 주최자 확인 인덱스

**개요**
콜백 없는 카카오페이 구조에서 **참여자 셀프 신고 + 주최자 수동 확인**의 2단계로 납부를 추적. 참여자별 상태: `미납` / `확인 대기` / `확인` (+반려).

##### 납부_확인 — 셀프 신고 대조·확정

주최자

**트리거 / 진입**

**User Flow**

2. 카카오페이 입금 내역과 수동 대조
3. 일치 → **확인 확정** → 참여자 상태 `확인` + 알림
4. 전원 `확인` → 마감 CTA 활성(정산_마감)

**분기 / 엣지**
- 확인 타임스탬프 기록 — 분쟁(정산_이의) 시 양측 근거

- 확인 지연 시 참여자 안내 문구 — 📝카피(화면 구현 단계에서 확정)

#### 회비_납부 — 참여자 납부 인덱스

**개요**
참여자(부원+공개 번개 비부원)가 자기 몫을 확인·검증하고 카카오페이로 납부한 뒤 셀프 신고를 남기는 흐름. **사후 정산**=이벤트 후 실지출 몫 납부 · **사전 걷기**=이벤트 전 예상 회비 선납(동일 메커니즘, 시점만 다름 — #10-1).

##### 납부_실행 — 확인·송금·셀프 신고

**행위자**
정산 참여자(부원 · 공개 번개 비부원 — 앱 계정만 전제)

**트리거 / 진입**
정산 요청 알림 · 이벤트 상세 · 홈 todo(홈 도메인 핸드오프)

**User Flow**
1. 정산 상세 — 내 몫(개인별 조정 반영)·납부 기한
2. **실지출 내역·영수증 열람**("왜 이 금액인가" 검증)
3. `[카카오페이로 납부]` 딥링크 → 외부 송금
4. 앱 복귀 → **`[납부했어요]` 셀프 신고** → 상태 `확인 대기`
5. 주최자 확인(납부_확인) → `확인` 완료

**분기 / 엣지**
- 카카오페이 미설치/딥링크 실패 → **폴백: 계좌번호 복사**(회원 도메인 send-money 스펙과 정합)
- 금액 이의 → 납부 전 이의_제기 분기
- 기한 경과 후 납부 허용(리마인드 연동)
- 송금만 하고 셀프 신고 잊음 → 재진입 시 신고 유도 배너

##### 반려_후_재신고 — 반려 대응

**행위자**
반려된 참여자

**트리거 / 진입**
납부_반려 알림 → 정산 상세

**User Flow**
1. 반려 안내 확인(사유 표시 — 있으면)
2-a. 실제 미송금이었다면 → 재송금 → 셀프 신고
2-b. **이미 보냈다면** → 재신고(송금 정보 메모 첨부 — 시각·금액) 또는 이의_제기
3. 상태 `확인 대기` 재진입 → 주최자 재대조

**분기 / 엣지**
- **재신고 증빙(확정)** — 송금 캡처·메모 선택 첨부 허용(의무 아님) — 주최자 재대조 용이
- **반복 한도 없음(#7 확정)** — 기록(타임스탬프)이 남는 것으로 충분, 이의는 언제든 병행 가능
- 반려 상태의 기한 처리(기한 경과로 취급하지 않음 권장)

##### 기한_연장_신청 — 참여자의 납부 기한 연장 요청

**개요**
기한 내 전액 납부가 어려운 참여자가 주최자에게 기한 연장을 신청하는 흐름. 부분 송금 상태를 만들지 않는 대신 도입된 메커니즘 — "덜 냈음"이 아니라 "늦게 냄"으로 관리한다.

**행위자**
정산 참여자(신청) · 주최자(승인/거절)

**트리거 / 진입**
정산 상세(참여자)의 `기한 연장 신청` — 기한 전·후 모두 가능

**User Flow**
1. 참여자: 연장 신청 → 희망 기한·사유(선택) 입력 → 제출 → 주최자 알림
2. 주최자: 승인(개인별 기한 갱신 → 리마인드 일정 재계산) / 거절(사유 회신)
3. 결과 알림 → 정산 상세에 개인 기한 표시

**분기 / 엣지**
- 연장은 개인 단위(전체 기한 아님 — 개인별 금액 수정과 같은 축)
- 연장 이력은 현황판·아카이브에 표시 — 완납 시 최종 상태는 동일하게 `확인`, 아카이브엔 `납부 완료 · 연장 N회(원기한→연장기한, 납부일)` 형식(크리틱5 ⑥ — 연장은 낙인이 아니라 과정 기록)
- **연장 승인 = 마감 연기 약속**: 연장 기한까지 강제 마감 불가(마감_확정 참조)
- 연장 반복 한도 없음(#7 결정 4와 일관) — 기록이 남는 것으로 충분

#### 정산_이의 — 분쟁 해결 인덱스

**개요**
금액·내역에 대한 참여자의 이의와 주최자의 처리. 셀프 신고·확인·반려의 타임스탬프가 양측 근거(원장 없는 구조의 분쟁 장치).

##### 이의_제기 — 참여자 이의

**행위자**
정산 참여자

**트리거 / 진입**
정산 상세의 `이의 제기` (납부 전·후 모두 가능)

**User Flow**
1. 사유 선택(금액 오류 · 내역 의문 · 냈는데 반려됨 · 중복 요청 등) + 상세 입력
2. 제출 → 주최자에게 알림 → 내 정산 상태에 `이의 진행 중` 표시
3. 처리 결과(정정/반려 회신) 알림 수신

**분기 / 엣지**
- **이의 중 납부 허용(#7 확정)** — 이의와 납부는 병행(먼저 내고 따지기 가능)
- 이의제기 진입 위치 재검토 — 📝UI 단계(#7 메모: "이의제기·신고의 위치가 이상함")
- 이의 철회(참여자 스스로 취소) 허용
- 마감 후 아카이브에서 발견한 오류의 이의 → 정정 기록 추가로 처리(확정, 마감_확정 참조)

##### 이의_처리 — 주최자 확인·정정

**행위자**
주최자

**트리거 / 진입**
이의 제기 알림 · 현황판의 이의 표시

**User Flow**
1. 이의 내용·근거(신고/확인/반려 타임스탬프) 확인
2-a. **인정** → 정산_수정으로 정정(금액 조정 → 재알림·차액 환불)
2-b. **반려** → 사유 회신 → 참여자 알림
3. 처리 결과가 정산 기록에 남음

**분기 / 엣지**
- **주최자 무응답 에스컬레이션** — ⚠팀논의(모임장 개입까지 vs 운영자 포함, 2026-07-11 보류)
- **반복 이의 제한 없음(#7 확정)**
- **이의·정정 이력의 아카이브 표시(확정)** — 아카이브_열람 참조

#### 정산_마감 — 확정·공개 인덱스

**개요**
전원 납부 확인 후 정산을 확정(잠금)하고 전 부원 열람 아카이브로 공개. 에스크로·원장 없는 구조에서 "검증 가능성"을 보증하는 마지막 단계. **사후 정산**=잉여 환급 없음 · **사전 걷기**=환급·추가정산 완료 후 마감(#10-1) `[크리틱4]`.

##### 마감_확정 — 정산 잠금

**행위자**
주최자

**트리거 / 진입**
납부 현황판 전원 `확인` 시 마감 CTA 활성

**User Flow**
1. 마감 요약 미리보기(실지출 합계·영수증·인별 납부 기록)
2. 되돌릴 수 없음 고지 → 마감 확정
3. 기록 잠금(읽기 전용) → 아카이브 공개 · 부원 알림(선택)

**분기 / 엣지**
- **강제 마감 허용(확정)** — 미납자가 남아도 마감 가능, `미납` 기록 잔존. 단 **연장 승인자의 연장 기한 전에는 마감 불가**(크리틱5 ⑥ — 주최자가 승인한 약속은 자기 구속, 기한 경과 후 일반 규칙 복귀)
- **마감 후 오류 → 별도 정정 기록 추가(확정)** — 원본은 잠금 유지, 정정 기록을 덧붙임(append). 아카이브에 정정 이력이 투명하게 표시(구 원장의 append-only 정신)
- 마감 기한 없이 방치된 정산(주최자 이탈) → 모임장 대리 마감(⚠논의)

##### 아카이브_열람 — 마감 기록 공개 열람

**행위자**
전 부원(비부원은 자신이 참여했던 정산만)

**트리거 / 진입**
지난_이벤트_열람(모임_홈) → 이벤트의 정산 기록 · 마감 직후 알림

**User Flow**
1. 정산 개요 — 이벤트명·주최자·마감일·참여 인원
2. 실지출 항목별 금액 + 영수증(탭 → 원본 확대)
3. 인별 납부 기록(금액·확인일·개인별 조정 내역)
4. 읽기 전용 잠금 표시(마감 후 수정 불가 = 신뢰 신호)

**분기 / 엣지**
- **인별 납부 금액 전부 공개(확정)** — 비참여 부원도 인별 금액·납부 상태 열람. "검증 가능성"이 제품 존재 이유
- **이의·정정 이력 표시(확정)** — 몇 차 정정·인정된 이의가 이력으로 남음
- **모임 장부 CSV(크리틱5 ⑤)** — 모임장·관리자가 기간 선택 → 전체 정산 장부 내보내기(승계 인수인계·회계 보고용, 개인 CSV와 대칭)
- 탈퇴자 기록 표기(유지 권장 — 아카이브 무결성)
