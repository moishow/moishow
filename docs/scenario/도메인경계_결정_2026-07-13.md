## 1. 정산 타이밍 → 주최자 선택 (사후 정산 / 사전 걷기) ⭐

**결정:** 이벤트 **개설 시 정산 방식을 주최자가 택1**한다.
- **사후 정산**: 이벤트 종료 후 실지출 기준으로 걷기. 환급·추가정산 없음. 노쇼=사후 페널티 청구.
- **사전 걷기**: 예상 회비 **선납** → 이벤트 후 실지출로 정산 → **잉여 환급 / 부족분 추가정산**. 노쇼 페널티(설정 시)는 선납서 차감해 실참여자 분담 경감, 나머지 환급(수금 확실).

**근거(재론, 2026-07-13):** 최초 "사후 단일" 추인은 성급했음. **노쇼 페널티(옵션)는 사후 청구 시 수금이 사실상 불가**(안 온 사람이 벌금을 나중에 보낼 인센티브 없음) — 사전 걷기(예약금)라야 선납분에서 페널티만큼 확실히 차감된다. 노쇼가 걸린 정모·예약형 번개가 많아 두 방식 모두 필요. 사전 걷기도 **에스크로/포인트 없이** 성립(주최자가 카카오페이로 선수령, 앱은 기록·체크만).

**모델 (두 방식 공통 원칙):**
- 앱은 돈을 거치지 않음(카카오페이 person-to-person) · 기록·확인·공개로 신뢰
- 정산 자체는 **실지출 기준 1/N**(개인별 조정 가능) — 방식 차이는 **걷는 시점 + 정산 후 차액 방향**
- 노쇼 페널티(**옵션** — 개설 시 설정): 노쇼자가 페널티 부담 → **페널티 수입은 실참여자 분담금에서 차감**해 참여자 부담 경감(크리틱5 ① 검증식). 사후=노쇼가 페널티 사후 납부 / 사전=노쇼 선납서 페널티만 차감(나머지 환급)

**파급 (문서 수정 목록):**
- 모임 `정모_개설`·`번개_개설`: **정산 방식 선택 필드**(사후/사전) + (사전) 예상 회비 입력 추가
- 모임 `정산_만들기`: 방식별 분기 — 사후=실지출 후 걷기 / 사전=선납→차액 환급·추가
- 모임 `노쇼_처리`: 페널티(옵션) 수입=실참여자 분담 경감 · 사후=사후 청구 / 사전=선납서 페널티만 차감(나머지 환급)
- 홈 `오늘_챙길_일`: `회비_송금_진입`(공통) + `추가정산_송금_진입`·`환급_수령확인`(**사전 걷기 이벤트에서 활성**) 존치
- 홈 `큐_정렬_갱신`·`알림`: 추가정산·환급 항목 **사전 이벤트에서 노출**
- 회원 `정산_내역`: 송금·수금·환불 내역 모두 표시(방식 무관)

**남는 질문(별건, 백로그):** 정기 회비(연/월회비 선수납)는 이벤트 정산과 다른 개념 — MVP는 "정모/번개 정산"만 커버, 정기 회비 모델링은 이후.

## 2. 무소속 번개 → 폐기

**결정:** 번개는 항상 모임 소속(모임⊃번개 엔티티 모델 유지). 무소속 번개(홈 제안본 P1) 폐기.

**근거:** 비부원 참여 수요는 **공개 번개**가 이미 커버(가입 유입 경로).

**파급:** 홈 `추천_번개_참여`를 "공개 번개 발견"으로 축소.

## 3. 탐색 소유 → 홈 도메인

**결정:** 모임·번개 탐색·검색은 **홈 도메인 소유**.

**근거:** 모임 도메인 경계에 이미 "탐색·검색=홈"으로 명시. 예경이 홈 오너라 자연 귀속.

**파급:** (기존 AI판) W-06 탐색 = 홈 소유로 확정. 홈 제안본이 탐색을 모임 소유로 본 표기 정정.

## 4. 신고 소속 → 운영자 신고함 통합 (7번 파급)

**결정:** 콘텐츠(게시글/댓글) 신고와 사람(사용자) 신고를 **운영자 신고함으로 통합** 처리.
- 1차 처리(숨김·삭제·경고): **모임 관리자** (F4 유지)
- 에스컬레이션·사람 신고·CS: **플랫폼 운영자** (7번 풀 모더레이션 채택으로 MVP 편입)

**근거:** 7번에서 운영자 처리 큐를 MVP로 넣기로 함 → #4의 "운영자 에스컬레이션=post-MVP"를 상향 조정하는 게 일관.

**⚠ 스코프 경보:** 운영자 **역할·콘솔**이라는 새 빌드 범위가 생김(운영 인력 전제). 부담 시 7번과 함께 재검토.

## 5. 가입 방식 정합 → 이원화 채택

**결정:** 모임 개설 시 선택하는 **즉시 가입 / 승인제 이원화**(모임 스펙) 채택.

**근거:** 확정 모델. 홈 제안본의 "신청서 항상 경유(A1)"는 **승인제에만** 적용, 즉시 가입 모임은 스킵.

**파급:** 홈 `태그_탐색`·`추천_모임_가입_진입` 문구를 이원화에 맞게 수정.

## 6. P3 번복(돈의 집) → 하이브리드

**결정:** 돈 **액션**(회비 송금·마감 todo)은 **홈**, 돈 **기록 열람**(정산 내역)은 **마이**로 분리 유지.
- 홈 = 현황·행동 (오늘 챙길 일·다가오는 일정)
- 마이 = 정체성·기록·설정 (정산 내역 조회·모임 레벨)

**근거:** "돈의 집=홈"으로 단일화하기보다, 액션/기록을 축으로 나누면 중복 없이 각자 역할이 선명(리뷰 결정 1과 절충). 중복 최소화 원칙 유지.

**⚠ 오너 확정:** 예경 P3(원안=마이에 돈 현황) 번복 사안 — **예경 confirm 필요.** 위는 해찬 담당 스펙 기준 제안.

## 7. 사용자 차단·신고(사람)·CS → 풀 모더레이션

**결정:** MVP에 **차단 + 사람 신고 접수 + 운영자 처리 큐** 전부 포함.
- 차단: 즉시 효과(DM·상호 노출 차단)
- 신고: 접수 → 운영자 신고함
- 운영자: 처리(경고·정지 등) — 4번과 통합 콘솔

**근거:** 개인 채팅 "시작 제한 없음"의 선행 안전장치. 접수만 하고 처리를 미루면 신고가 방치됨.

**파급:** 회원 도메인에 **`차단·신고` 시나리오 신설**. 운영자 역할/콘솔 신규 스펙 필요(4번과 공유). 개인_채팅·모임/이벤트 채팅의 "차단·신고" 연동 지점(팀회의 항목이었던 것) 해소.

**⚠ 스코프 경보:** 운영 인력·운영자 콘솔 전제 — 4번과 동일.

## 8. 정산 명단 실명 → 닉네임 통일 표시

**결정:** 정산·명단·채팅 전부 **모임별 닉네임**으로 통일 표시. 실명은 본인에게만.

**근거:** 리뷰 결정 3 + 닉네임 통일 원칙(SCENARIO §4 모임_홈_열람)과 이미 일치 — 사실상 기확정 확인.

**파급:** 없음(현행 유지). 동명이인 혼동은 화면 단계에서 "닉네임(실명 이니셜)" 병기 옵션만 검토(📝UI 이연).
